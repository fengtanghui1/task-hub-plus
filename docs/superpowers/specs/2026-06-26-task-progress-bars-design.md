# Task Progress Bars Design

Date: 2026-06-26
Status: Draft for review

## 背景

Task Hub 已经支持识别 Markdown 任务层级，并在任务视图中展开/收起子任务，但当前父任务只展示标题和完成状态，用户无法快速判断一个任务树整体做到了什么程度。

这会带来几个实际问题：

- 父任务下面有很多子任务时，必须展开后逐条看，才能判断进度。
- 多层级子任务场景下，用户很难在列表里快速比较“哪个项目更接近完成”。
- 父任务被手动勾选或未勾选时，不一定能反映其子任务树的真实完成情况。

目标是在不引入复杂任务系统、不修改 Markdown 语法的前提下，为已有的层级任务增加一个轻量、稳定、可关闭的“子任务进度条”。

## 目标

- 让用户在任务列表中一眼看出带子任务的父任务完成进度。
- 支持多层级子任务的递归进度计算。
- 提供简单明确的设置项，允许用户全局开启或关闭该能力。
- 保持现有任务树、筛选、展开/收起、完成写回逻辑不变。

## 非目标

- 不新增新的 Markdown 任务语法。
- 不自动改写父任务的完成状态。
- 不为没有子任务的普通任务显示进度条。
- 不在第一版覆盖所有视图；优先覆盖任务视图，其他视图后续再评估。
- 不把该功能扩展为“项目管理”或“统计分析”系统。

## 用户问题

用户需要的不是“又一个百分比”，而是一个能在扫描任务列表时立即回答下面问题的信号：

- 这个父任务现在大概做到了多少？
- 这个子树里还有多少未完成工作？
- 哪些任务值得我优先展开查看？

因此，这个进度条应当是任务行上的辅助信息，而不是替代 checkbox 的主状态。

## 方案选项

### 方案 A：只在任务视图显示父任务进度条

推荐方案。

做法：

- 仅在 `Tasks` 视图的任务列表行中，为“存在子任务的任务”显示进度条和百分比。
- 在任务详情面板中补充相同进度信息，帮助用户理解计算结果。
- 设置页提供一个全局开关，默认开启。

优点：

- 改动范围小，和现有任务树结构最贴合。
- 用户最常在任务视图里处理层级任务，收益最大。
- 不会同时把 Calendar / Tags / 外部任务源一起卷进来，风险可控。

缺点：

- 标签视图或日历视图里暂时看不到同样的进度提示。

### 方案 B：在所有展示任务的视图里统一显示进度条

做法：

- 任务视图、标签视图、日历详情、可能的弹窗详情都显示相同进度条。

优点：

- 视觉和信息一致性最好。

缺点：

- 需要逐个检查各视图的数据来源与层级关系。
- 日历和标签场景中并不总是完整展示任务树，容易出现“为什么这里是 50%，那里没显示”的困惑。
- 第一版实现、测试、文案成本都更高。

### 方案 C：只显示文本百分比，不显示条形进度

做法：

- 对父任务显示 `67%` 或 `2/3`，不显示条形条。

优点：

- 最省空间，样式成本最低。

缺点：

- 扫描效率明显差于条形可视化。
- 用户已经明确提出“进度条”，文字弱化了目标价值。

## 推荐方案

采用方案 A。

第一版聚焦任务视图中的层级 Markdown 任务，提供：

- 任务行上的细进度条
- 旁边的百分比数字
- 任务详情中的进度说明
- 设置页全局开关

这样既满足核心诉求，也能把数据规则先定义清楚，为后续扩展到标签视图或日历详情留出空间。

## 功能概述

当一个任务存在一个或多个子任务时，Task Hub 默认在该任务上显示“子任务进度条”。

进度条用于表示该任务子树的完成进度，而不是该任务本身 checkbox 的状态。没有子任务的任务不显示进度条。

用户可以在设置中关闭该能力。关闭后：

- 任务行不显示进度条和百分比。
- 任务详情不显示子任务进度模块。
- 任务的展开/收起、完成状态、筛选逻辑不受影响。

## 范围

第一版范围：

- Vault Markdown 任务。
- 已有层级关系的子任务树。
- 已启用的 linked-note subtasks，如果它们已经作为父任务的子任务接入当前任务树。
- 任务视图列表行。
- 任务详情面板。
- 设置页全局开关。

不在第一版范围：

- Apple Reminders / Dida 等外部任务源的层级进度。
- Calendar 视图卡片上的进度条。
- Tags 视图卡片上的进度条。
- 基于进度的排序、筛选或统计。

## 交互设计

### 设置项

在设置页的基础功能区域增加一个开关：

- 名称：`Show subtask progress bars`
- 中文：`显示子任务进度条`
- 默认值：开启

说明文案建议：

- 英文：`Show a progress bar on tasks that contain subtasks. Progress is calculated from the full subtask tree.`
- 中文：`为包含子任务的任务显示进度条。进度会根据完整子任务树自动计算。`

该设置是纯展示开关，不改变任务数据本身。

### 任务列表

当任务存在子任务时：

- 在任务正文下方显示一条细进度条。
- 在进度条右侧显示整数百分比，例如 `0%`、`50%`、`100%`。
- 进度条应在视觉上弱于任务标题和 checkbox，避免抢主操作。

当任务没有子任务时：

- 不显示进度条。

当任务被折叠时：

- 仍然显示进度条。
- 进度继续按完整子树计算，不因子任务当前不可见而变化。

### 任务详情

在详情面板中，当任务存在子任务时增加一个轻量进度模块：

- `Subtask progress / 子任务进度`
- 百分比
- 已完成子项数与总子项数摘要

例如：

- `50%`
- `1 of 3 direct children complete-equivalent`

这里的“complete-equivalent”表示包含递归折算后的子项贡献，不强制把说明直接写给用户，但产品语义应保持一致。

## 进度计算规则

### 基本规则

1. 没有子任务的任务不显示进度条。
2. 有子任务的任务，进度条范围是 `0%` 到 `100%`。
3. 直接子任务若没有自己的子任务：
   - 已完成记为 `100%`
   - 未完成记为 `0%`
4. 直接子任务若有自己的子任务：
   - 先递归计算该子任务的进度
   - 再把该进度作为该子任务对父任务的贡献
5. 父任务最终进度 = 所有直接子任务贡献值的平均值。

公式：

```text
progress(task) =
  average(
    progress(child1),
    progress(child2),
    ...
  )

leaf child progress =
  completed ? 100 : 0
```

### 递归示例

```text
Release Task Hub
├── Refresh docs               [x]
├── QA pass
│   ├── Manual smoke           [x]
│   └── Regression suite       [ ]
└── Publish release            [ ]
```

计算过程：

- `Refresh docs` = `100%`
- `QA pass` = `(100 + 0) / 2 = 50%`
- `Publish release` = `0%`
- `Release Task Hub` = `(100 + 50 + 0) / 3 = 50%`

### 父任务完成状态与进度条的关系

进度条反映的是“子任务树完成度”，不是父任务自身 checkbox 状态。

因此，如果父任务被手动勾选，但其子任务仍未全部完成：

- 父任务 checkbox 仍显示已完成
- 进度条仍显示真实的子任务完成比例，而不是强行显示 `100%`

这样可以避免父任务状态掩盖子任务仍未完成的事实。

### 筛选状态与进度条的关系

为了避免用户切换筛选器后同一个任务的进度突然变化，进度计算应基于该任务的完整已索引子任务树，而不是仅基于当前可见行。

这意味着：

- 关闭“显示已完成任务”后，父任务进度不应因为完成项被隐藏而从 `50%` 变成 `0%`
- 文本/标签/来源筛选也不应重算同一个任务的基础进度

显示范围由当前视图决定，进度计算基线由完整任务树决定。

## 显示规则

- 只对至少拥有一个子任务的任务显示进度条。
- 进度条显示整数百分比，默认四舍五入。
- `0%` 和 `100%` 都应正常显示。
- 完成的父任务仍可显示进度条，因为它传达的是子树状态。
- 外部任务若当前没有层级关系数据，则不显示进度条。

## 边界情况

### 只有一个子任务

父任务进度等于该唯一子任务的进度。

### 子任务全部完成

父任务显示 `100%`。

### 子任务全部未完成

父任务显示 `0%`。

### 多级嵌套很深

仍按递归规则计算，但实现上应避免在每一行重复深度遍历；建议在单次渲染前预计算并缓存。

### 过滤后父任务不可见、子任务单独可见

如果某个子任务因筛选结果单独显示，而它自己也拥有子任务，则它仍显示自己这棵子树的进度。

### linked-note subtasks 开关关闭

如果 linked-note subtasks 没有接入任务树，则它们不参与进度计算。

## 文案与信息架构

建议新增文案键：

- `showSubtaskProgressBars`
- `showSubtaskProgressBarsDesc`
- `subtaskProgress`
- `subtaskProgressSummary`

中英文都需要同步补齐。

## 样式方向

- 进度条使用细条样式，避免把任务行做成项目管理卡片。
- 宽度跟随任务正文区域，不单独占据一整行大块空间。
- 颜色优先复用 Obsidian 主题变量或当前任务来源色的低对比版本。
- 完成父任务的进度条可以保留，但整体降低对比度，以避免与 checkbox 状态冲突。

建议结构：

```text
[ ] Release Task Hub                 50%
    Refresh docs, QA, publish
    [██████████----------]
```

## 数据与实现建议

这一节是为了保证设计可落地，不等于进入实现。

- 继续复用现有 `parentId` / `indent` 层级关系。
- 在任务视图渲染前建立 `taskId -> childTasks[]` 映射。
- 基于完整任务树预计算 `taskId -> progressInfo`：
  - `hasChildren`
  - `percent`
  - `directChildCount`
  - `completedLeafEquivalent` 或等价摘要字段
- 任务行和详情面板只消费预计算结果，不在 DOM 层临时递归计算。

## 验收标准

- 默认情况下，存在子任务的任务会显示进度条与百分比。
- 在设置中关闭“显示子任务进度条”后，任务行和详情中的进度模块都消失。
- 没有子任务的任务不会显示进度条。
- 单层子任务的完成状态能正确映射为父任务 `0%` 到 `100%` 进度。
- 多层子任务会按递归规则正确计算。
- 任务折叠时，父任务进度不丢失。
- 隐藏已完成任务或应用其他筛选时，同一任务的基础进度不会因为可见性变化而被错误重算。
- 父任务即使已勾选完成，只要其子任务未全部完成，进度条仍显示真实子树进度。

## 风险与取舍

- 风险：父任务已完成但进度条低于 `100%`，部分用户可能觉得矛盾。
  - 取舍：优先保持“子任务树真实完成度”语义，避免误导。

- 风险：若未来外部任务源支持层级关系，用户会期待一致体验。
  - 取舍：第一版先把 Vault 任务规则做清楚，再决定是否扩展到 Apple Reminders / Dida。

- 风险：如果直接按当前可见列表计算，进度会在筛选后跳变。
  - 取舍：明确要求按完整任务树计算，保证稳定性。
