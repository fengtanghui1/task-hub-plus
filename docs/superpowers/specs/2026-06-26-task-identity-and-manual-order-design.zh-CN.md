# Task Hub 任务稳定身份与手动排序设计

日期：2026-06-26
状态：设计冻结前草案，供后续实现与外部源扩展共用

## 目标

Task Hub 需要支持两类长期存在且会随版本迭代扩展的能力：

1. 列表视图中，同一日期内任务的手动显示顺序。
2. 后续持续接入新的外部任务源，而不破坏已有排序、跳转、写回与数据兼容。

为了让这两类能力长期稳定，本设计定义一个统一的“任务稳定身份”模型，并明确：

- 什么数据应存入插件本地数据；
- 哪些数据不应写回用户 Markdown 原文；
- 不同来源的任务如何映射到统一身份；
- 如何控制 `data.json` 的长期增长；
- 当本地排序/身份数据损坏、失配或过大时如何降级和恢复。

本设计是后续任务排序、任务身份、外部源扩展的基础协议。后续 agent 若修改这些主题，必须先读取本文件。

## 背景问题

当前 vault 任务的 `TaskItem.id` 来自：

- `filePath`
- `line`
- `rawLine hash`

这类 ID 适合“安全写回原文”，因为它紧贴当前源文件中的具体行。但它不适合“长期身份”场景：

- 用户在笔记中插入几行，上下文变化，行号会变；
- 用户在 Task Hub 或编辑器中修改标题、标签、日期，`rawLine hash` 会变；
- 未来同一任务可能会被发送到外部源、迁移来源、再同步回来。

如果手动排序直接依赖当前 `TaskItem.id`，那么排序映射会很容易失效。

## 设计原则

### 1. 分离“当前位置身份”和“长期身份”

保留现有 `TaskItem.id` 作为运行时定位与安全写回使用的“当前位置身份”。

新增 `stableId` 作为：

- 列表手动排序；
- 任务关系延续；
- 外部源统一映射；
- 后续本地视图状态关联

使用的“长期身份”。

这两个身份解决不同问题，不应合并。

### 2. 不要求用户维护任何身份字段

用户不应：

- 在任务行中手动写 ID；
- 维护外部映射；
- 处理排序 JSON；
- 理解内部 stableId 的具体格式。

所有 stableId 的分配、继承、迁移、清理都必须由插件自动完成。

### 3. 不默认污染用户 Markdown 原文

第一版稳定身份方案不向 vault 任务正文写入隐藏元数据，不增加 `%%...%%`、HTML 注释或额外 frontmatter。

理由：

- 会污染用户笔记；
- 会增加写回复杂度；
- 会和其他插件、用户编辑习惯冲突；
- 会扩大冲突面。

除非未来明确决定引入“正文内嵌持久标识”，否则 stableId 仅保存在插件数据中。

### 4. 外部源必须走统一命名空间协议

后续新增外部任务源时，不允许每个源自己发明排序键格式。所有外部源都必须通过统一 stableId 命名空间接入。

## 术语

### 当前位置身份

运行时用于定位当前任务源位置的 ID，例如当前的 `TaskItem.id`。

用途：

- 找回源文件任务；
- 安全写回；
- 检测是否还是同一行。

它可以变化，也允许变化。

### 稳定身份 stableId

插件内部长期使用的任务身份。

用途：

- 手动排序映射；
- 本地关系延续；
- 视图状态关联；
- 外部源兼容。

它应在“同一任务小幅修改、位置变化、同源再扫描”场景下尽量保持不变。

## stableId 设计

### vault 任务

vault 任务的 stableId 由插件生成并持久化，例如：

```text
vault:th_20260626_ab12cd34
```

要求：

- 对用户不可见；
- 与当前 `TaskItem.id` 解耦；
- 一旦分配，后续尽量继承，不重新生成。

### 外部任务

外部任务直接使用其来源系统的稳定主键，外面包一层 Task Hub 命名空间。

建议格式：

- Apple Reminders：`apple-reminders:{externalId}`
- Dida：`dida:{externalId}`
- 未来 Google Tasks：`google-tasks:{externalId}`
- 未来 Microsoft To Do：`microsoft-todo:{externalId}`

规则：

- 命名空间必须稳定、可读、不可与其他来源混淆；
- 不允许省略来源前缀；
- 不允许直接裸存第三方 ID。

## 数据模型

### TaskItem 扩展

运行时任务对象应扩展出 stableId：

```ts
type TaskItem = {
  id: string;
  stableId: string;
  ...
}
```

其中：

- `id` 继续负责当前位置定位；
- `stableId` 负责长期映射。

### 插件设置/数据中的排序映射

列表同日期手动排序只存“覆盖数据”，不存全量任务快照。

建议结构：

```ts
type TaskListManualOrder = Record<string, string[]>;
```

示例：

```json
{
  "taskListManualOrder": {
    "2026-06-27": ["vault:th_a1", "apple-reminders:123", "dida:456"]
  }
}
```

含义：

- key 是日期；
- value 是该日期下按显示顺序排列的 stableId 列表；
- 未出现在列表中的任务，渲染时回退到默认顺序。

## stableId 继承策略

### 1. Task Hub 内部编辑优先显式继承

如果任务修改由 Task Hub 自己发起，例如：

- 改标题；
- 改标签；
- 改日期；
- 改提醒时间；
- 发送到可写外部源后触发本地替换/迁移；

则插件已经知道“旧任务”和“新任务”的关系，不应再依赖模糊匹配。

要求：

- Task Hub 内部写回成功后，直接把旧 stableId 迁移给新任务；
- 若发生来源迁移，保留 stableId 延续规则，或按明确迁移策略处理；
- 不允许在 Task Hub 内编辑后重新生成 stableId。

这是最准确、最稳定的继承路径。

### 2. 编辑器外部修改回退到索引层自动匹配

当用户直接在 Obsidian 编辑器中修改任务，或外部源同步回来产生变化时，插件只能在重新索引时自动判断“这是否还是原任务”。

推荐在 `taskIndex` 层做旧任务与新任务的匹配，按从强到弱的顺序继承 stableId。

### 3. 推荐匹配特征

对 vault 任务，候选匹配应至少考虑：

- `filePath`
- 归一化任务文本
- `heading`
- `indent`
- `parent` 关系
- `dueDate`
- 标签集合
- 原行号距离

推荐策略：

1. 先在同文件内匹配；
2. 先找文本完全相同、日期相同、层级接近的候选；
3. 若多个候选都像，选旧行号距离最近的；
4. 若置信度不足，宁可判为新任务，也不要错误继承 stableId。

目标是“误丢排序”优先于“错误把两个不同任务并成一个”。

## 手动排序渲染规则

同一日期内渲染任务时：

1. 先收集该日期下所有任务；
2. 读取 `taskListManualOrder[dateKey]`；
3. 按 stableId 顺序排列命中的任务；
4. 未命中的任务按原默认顺序追加在后；
5. 已在排序表中但当前日期不存在的 stableId 不应参与渲染。

这样可以保证：

- 手动顺序只覆盖用户实际调整过的部分；
- 新任务自然落到尾部；
- 删除任务后不会阻塞渲染。

## data.json 增长控制

`data.json` 不能无限累积历史排序垃圾。必须把它当成一个需要清理的缓存型数据，而不是永久日志。

### 必须执行的清理规则

1. 删除空日期条目。
   - 如果某个 `dateKey` 的排序数组为空，立即移除这个 key。

2. 删除已回到默认顺序的日期条目。
   - 若某日期当前任务按默认顺序与手动顺序完全一致，则移除该 key。

3. 删除找不到对应任务的孤儿 stableId。
   - 若某 stableId 在当前索引/同步结果中已不存在，则从排序数组移除。

4. 控制无命中日期的保留时间。
   - 若某日期长期没有任何任务命中，可按策略删除，例如只保留近期窗口内的空映射，或在启动时清掉全部无命中日期。

### 不建议做法

- 不要存每次拖拽历史日志；
- 不要存完整任务副本；
- 不要为每个任务长期保留大对象元数据；
- 不要让排序映射变成“所有看过的任务全量镜像”。

## data.json 脏数据与失败恢复

本地数据不是可信源，只能视为“可丢失但可重建的辅助状态”。因此排序/稳定身份实现必须把恢复能力设计进去。

### 读取失败时的总原则

1. 不能因为排序/稳定身份数据损坏而阻塞插件启动。
2. 不能因为单个字段脏掉就丢弃整个插件设置。
3. 不能在无法确认正确性的情况下把脏数据继续写回扩散。

### 建议的失败处理策略

1. `loadData()` 成功但结构不合法：
   - 只丢弃不合法的排序/身份字段；
   - 其他已知合法设置继续保留；
   - 将脏字段回退到默认值；
   - 记录一次 notice 或内部日志，说明排序缓存已重置。

2. 某个日期的排序数组不合法：
   - 例如不是数组、元素不是字符串、数组过长、出现重复值；
   - 只清理该日期条目，不影响其他日期。

3. 某个 stableId 不合法：
   - 例如空字符串、缺少来源命名空间、明显不是当前协议格式；
   - 直接移除该 stableId；
   - 如果移除后该日期为空，则删除整个日期条目。

4. stableId 继承映射失败：
   - 宁可给 vault 任务重新分配新的 stableId，也不要错误继承；
   - 后果应限制为“该任务手动排序丢失”，而不是污染其他任务顺序。

### 脏数据判定最小规则

以下数据应视为脏数据并在归一化阶段清理：

- `taskListManualOrder` 不是对象；
- `dateKey` 不是 `YYYY-MM-DD`；
- value 不是字符串数组；
- 数组中存在重复 stableId；
- stableId 不是已知命名空间格式；
- 某日期排序数组长度远超该日期实际任务量；
- 排序数组仅包含孤儿 stableId。

### 写回时的安全策略

每次保存排序/身份数据前，先做一次轻量归一化：

1. 去重；
2. 移除非法 stableId；
3. 移除空日期；
4. 移除回到默认顺序的日期；
5. 再写入 `data.json`。

这一步必须在保存前做，而不是只依赖启动时清理。

## data.json 文件大小控制

排序/稳定身份数据必须有明确的尺寸控制策略，不能假设“以后再清理”。

### 总原则

1. 本地身份/排序数据是缓存，不是审计日志。
2. 可重建数据优先瘦身，不保历史。
3. 默认实现应偏向删除旧覆盖数据，而不是无限保留。

### 推荐的大小控制策略

1. **按需存储，而不是全量镜像**
   - 只存用户实际手动调整过的日期；
   - 不存未调整日期；
   - 不存任务全文、标题、标签快照。

2. **限制单日期数组大小**
   - 单个 `dateKey` 的排序数组不应无限增长；
   - 超出合理阈值时，应按当前日期实际任务集裁剪；
   - 不允许保留明显多于该日期任务数的历史 stableId。

3. **限制历史日期窗口**
   - 对长期无命中的日期条目执行清理；
   - 可以采用“只保留近期窗口 + 当前仍命中的日期”的策略；
   - 排序是视图辅助状态，不应永久记住几年前的一次拖拽。

4. **在保存时做压缩**
   - 清理重复项；
   - 清理默认顺序项；
   - 清理孤儿项；
   - 清理空对象。

### 建议的实现预算

不强制某个精确字节数，但建议实现时设定软预算，例如：

- 排序覆盖表应长期保持在“远小于整个 settings 主体”的量级；
- 一旦发现排序数据增长异常，应优先触发压缩而不是继续累积；
- 如果未来需要更精确控制，可加入内部统计字段或调试命令，但不要把统计信息长期写入用户数据。

### 超预算时的降级策略

如果检测到排序数据异常膨胀，推荐按以下优先级删减：

1. 先删空日期；
2. 再删无命中日期；
3. 再删回到默认顺序的日期；
4. 最后按最旧/最不活跃日期裁剪。

不允许因为超预算而删除：

- 用户基础设置；
- 当前仍命中的外部源关键配置；
- 任务写回、安全定位所必需的运行时状态。

## 外部源兼容协议

后续新增外部任务源时，必须满足下面 4 条：

1. 能提供来源稳定主键。
   - 如果外部源连稳定 `externalId` 都没有，需要先单独设计来源级身份策略，不可直接接入手动排序体系。

2. 必须映射到命名空间 stableId。
   - 例如 `provider-name:{externalId}`。

3. 来源层变更不得破坏排序层接口。
   - 列表排序逻辑只依赖 stableId，不应知道每个 provider 的内部细节。

4. 同步后必须触发孤儿清理。
   - 如果外部任务被删除或来源断开，排序表中的对应 stableId 要能被清理。

## 文档用途

本文件的作用不是实现说明，而是“稳定协议定义”。它主要约束：

- stableId 的职责边界；
- 外部源 stableId 命名空间；
- Task Hub 内编辑与外部编辑的继承规则；
- 手动排序数据结构；
- 数据清理、脏数据恢复与增长控制。

后续如果出现以下需求，必须先读本文件，再改代码：

- 同日期拖拽排序；
- 新外部任务源接入；
- 任务发送/迁移后身份延续；
- `data.json` 中排序/身份数据结构调整；
- 任务身份稳定性相关 bug 修复。

## 当前非目标

本设计当前不覆盖：

- 通过拖拽修改源 Markdown 中任务行的真实上下顺序；
- 在 Markdown 原文中嵌入 stableId；
- 跨日期排序与日历排序统一协议；
- 事件（calendar event）与任务共用同一手动排序表。

这些都需要独立设计，不应在本协议里顺手扩展。
